Software Requirements Specification (SRS)
EEC B2C Platform

Version: 1.0
Date: February 28, 2026
Document Type: Software Requirements Specification

1. Introduction

1.1 Purpose

This SRS defines the functional and non-functional requirements of the EEC B2C platform, a role-based EdTech web application for students, teachers, administrators, and public users. It serves as a baseline for development, testing, and maintenance.

1.2 Scope

EEC B2C provides authentication, academic content management, question bank operations, exam and attempt flows, AI-powered question generation, self-study, paid study materials, subscriptions, wallet and redemption capabilities, notifications, real-time chat, and admin-level CMS and promotion management.

1.3 Intended Audience

Product owners, project managers, developers, testers, DevOps engineers, academic operations stakeholders, and support teams.

1.4 Definitions

TermDefinition
EEC B2CBusiness-to-consumer educational platform for learning, assessments, and digital product sales.
RBACRole-Based Access Control across Public, Student, Teacher, and Admin.
StageLearning progression level unlocked based on purchases/subscriptions.
Self StudyAI-generated question practice flow from user-uploaded PDFs.

2. Overall Description

2.1 Product Perspective

The system is a MERN-style web platform with React frontend, Node.js/Express backend, MongoDB database, Socket.IO for realtime communication, and external integrations for payment, media storage, push notifications, and AI services.

2.2 User Classes and Characteristics

User ClassDescriptionPrivileges
PublicUnauthenticated visitorView marketing content, browse packages, subscribe to newsletter
StudentLearner accountTake exams, buy materials/packages, use self-study, view results, chat
TeacherAcademic contributor accountStudent views, subject/topic operations, verified material upload, chat
AdminPlatform operatorFull CRUD and governance across all modules

2.3 Operating Environment

  • Frontend: React, Vite, TailwindCSS, Axios, React Router
  • Backend: Node.js, Express, Mongoose, JWT, Socket.IO
  • Database: MongoDB
  • Integrations: Razorpay, Cloudinary, Web Push (VAPID), Email, AI service
  • Deployment profile: browser clients + HTTP API server + WebSocket server

2.4 Constraints

  • JWT-based authenticated APIs must enforce role-specific authorization.
  • Payment verification must use Razorpay signature validation.
  • Sensitive routes (auth) are rate-limited and protected with security middleware.
  • CORS restrictions must permit only configured origins.

3. External Interface Requirements

3.1 User Interface

  • Public pages: home, feature pages, package browsing, static informational content.
  • Authenticated dashboard: role-sensitive navigation and module access.
  • Question management UIs for multiple question types (MCQ, essay, cloze, matrix, etc.).
  • Material browsing, purchase, and secure PDF viewing interfaces.

3.2 Software Interfaces

  • Payment API: Razorpay order creation and signature verification.
  • Storage API: Cloudinary media uploads.
  • Push API: Web Push subscription lifecycle and dispatch.
  • AI Service API: PDF text-based question generation.
  • Email API: transactional emails for key user flows.

3.3 Communication Interfaces

  • HTTPS REST APIs under /api/*.
  • Socket.IO channel for global chat and read-status events.
  • Webhook endpoint for Razorpay event processing.

4. System Features and Functional Requirements

4.1 Authentication and Account Management

  • FR-1: The system shall allow user registration and login with secure password handling.
  • FR-2: The system shall provide JWT-based session validation and current-user retrieval.
  • FR-3: The system shall support forgot-password and tokenized password reset.
  • FR-4: The system shall enforce route guards for authenticated and role-specific access.

4.2 Academic Master Data

  • FR-5: The system shall support admin CRUD for boards and classes.
  • FR-6: The system shall support subject/topic management with board/class mapping.

4.3 Question Bank and AI Questioning

  • FR-7: The system shall support CRUD for multiple question types.
  • FR-8: The system shall provide metadata APIs for class, stage, subject, and topic filtering.
  • FR-9: The system shall allow admin to upload PDF and generate questions via AI service.
  • FR-10: The system shall allow generated questions to be reviewed and optionally persisted.

4.4 Exams, Attempts, and Results

  • FR-11: Students shall be able to start and submit exams.
  • FR-12: The system shall persist attempts and provide user-wise result retrieval.
  • FR-13: Teachers/admins shall be able to view attempt analytics and detailed results.
  • FR-14: The system shall expose class rank and leaderboard views.

4.5 Syllabus and Stage Unlocking

  • FR-15: Stage 1 shall be unlocked by default for students.
  • FR-16: Additional stages shall be unlocked based on active package/subscription conditions.

4.6 Study Materials and Commerce

  • FR-17: Teachers/admin shall upload and manage paid/free study materials.
  • FR-18: Students shall browse materials filtered by class and board.
  • FR-19: The system shall enforce secure material access based on free/purchased status.
  • FR-20: The system shall support Razorpay and wallet-based purchase flows.
  • FR-21: The system shall generate and store purchase/invoice records.

4.7 Packages and Subscriptions

  • FR-22: Admin shall manage package catalog (CRUD and activation).
  • FR-23: Users shall subscribe via free or paid flows.
  • FR-24: The system shall track subscription lifecycle and allow cancellation.
  • FR-25: Admin shall view all subscription reports.

4.8 Wallet, Coins, and Gift Cards

  • FR-26: The system shall maintain wallet/points balances per user.
  • FR-27: Users shall redeem points to wallet or gift card where eligible.
  • FR-28: Admin shall manage gift card inventory and redemption statuses.

4.9 Notifications, Push, and Chat

  • FR-29: Admin shall create role-targeted notifications.
  • FR-30: Users shall fetch, read, and clear notifications.
  • FR-31: The system shall support push subscribe/unsubscribe and payload dispatch.
  • FR-32: The system shall support global chat with message persistence and admin moderation.

4.10 Self Study and Mind Training

  • FR-33: The system shall generate self-study questions from uploaded PDF content.
  • FR-34: The system shall evaluate self-study submissions and store scores/statistics.
  • FR-35: The system shall store mind-training game result submissions.

4.11 CMS and Promotion Engine

  • FR-36: Admin shall manage public site content sections (hero, features, about, office, etc.).
  • FR-37: The system shall run scheduled auto-promotion and support manual admin promotion operations.

5. Non-Functional Requirements

5.1 Security

  • NFR-1: Passwords shall be stored as salted hashes.
  • NFR-2: Protected APIs shall require valid JWT and role checks.
  • NFR-3: Security middleware (Helmet), CORS policy, and auth rate limits shall be enabled.
  • NFR-4: Payment confirmation shall require cryptographic signature validation.

5.2 Performance

  • NFR-5: Common API operations should return within acceptable interactive latency under normal load.
  • NFR-6: Chat event propagation should provide near real-time delivery for active users.

5.3 Reliability and Availability

  • NFR-7: Service health endpoint shall expose runtime availability state.
  • NFR-8: Scheduled promotion and keep-alive routines shall execute reliably after server startup.

5.4 Scalability

  • NFR-9: Modular route architecture shall support independent extension of functional domains.
  • NFR-10: Database schema shall support growth across users, attempts, purchases, and notifications.

5.5 Maintainability

  • NFR-11: The codebase shall keep domain separation through dedicated route/service/model modules.
  • NFR-12: Configuration shall remain environment-driven for deployability across environments.

7. Assumptions and Dependencies

  • MongoDB instance and connection credentials are available at runtime.
  • Razorpay, Cloudinary, push keys, mail, and AI service credentials are configured in environment variables.
  • Client origins in CORS settings are kept in sync with deployment domains.

8. Acceptance Criteria (Summary)

  • All core role-based user journeys (Public, Student, Teacher, Admin) are executable end-to-end.
  • Payment and access control logic correctly gates premium materials and stages.
  • AI question generation, self-study scoring, and attempts/results flows operate correctly.
  • Notification/chat features deliver and persist expected communication records.
  • Admin can manage academic masters, content settings, subscriptions, and promotions.

9. Traceability Baseline

The requirements above map to implemented backend route groups including (non-exhaustive): /api/auth, /api/users, /api/questions, /api/exams, /api/attempt, /api/study-materials, /api/packages, /api/subscriptions, /api/gift-cards, /api/notifications, /api/push, /api/chat, /api/ai-questions, /api/self-study, and /api/promotion.
